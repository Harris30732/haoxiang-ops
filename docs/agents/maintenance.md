# 制度維護協議（maintenance.md）

> 讀者：未來要修改制度檔（`CLAUDE.md`、`docs/agents/*`、`.claude/agents/*`）的 Claude。
> 這套制度 2026-07-03 由一個高階模型 session 建立；維護它的會是較小的模型。
> 所以預設立場是**保守**：能新增就不要改寫，能改寫就不要刪除。

## 1. 權限分級

### 不用問，直接做
- 在 `lessons.md` 追加踩坑紀錄（格式見 §3）。
- 修正客觀錯誤：路徑失效、檔名改了、指令改了（改之前先實際驗證新值是對的）。
- 在 `templates.md` 新增模板（不動既有模板）。
- 在 `docs/agents/reports/` 落研究報告。

### 要先用 AskUserQuestion 問過使用者才能做
- 修改或刪除 CLAUDE.md 的「硬規則」任何一條。
- 修改 dispatch.md 的升降級路徑、judgment.md 的完成判準／QG rubric（放寬尤其要問）。
- 刪除任何制度檔、或刪 `legacy/`、`docs/agents/backup/` 裡的備份。
- 修改 `.claude/agents/verifier.md`（驗收員被弱化＝整套驗證失效）。

### 永遠不准
- 因為「這條規則讓我現在的任務變麻煩」而當場改規則。規則擋路時：照規則做，
  或問使用者；事後覺得規則不對，記進 lessons.md 讓使用者裁決。
- 無備份改寫。

## 2. 修改流程（每次都一樣）

1. 備份：`cp {檔案} docs/agents/backup/{檔名}.{YYYY-MM-DD}.bak`（同日多次改，第一次備份即可）。
2. 改（新內容盡量用新增小節，而不是改寫舊句子）。
3. 自查 read-back：重讀整份，確認沒有跟其他制度檔矛盾——特別檢查
   CLAUDE.md 硬規則 vs dispatch.md vs judgment.md 三者的數字與路徑一致。
4. commit：`docs: 制度更新 — {一句話} [QG:分數]`，body 寫「為什麼改」。push。

## 3. 踩坑教訓寫回哪裡

| 教訓類型 | 寫到哪 |
|----------|--------|
| 環境/工具的坑（指令失敗原因、權限、MCP 行為） | `lessons.md` 追加一條 |
| 專案知識的坑（某檔案其實負責什麼、隱藏耦合） | `lessons.md`；若是長期事實，提議寫進 CLAUDE.md（走 §1 分級） |
| 制度本身的缺陷（規則互相打架、判準不管用） | `lessons.md` 標 `[制度]`，下次使用者在場時提出 |

`lessons.md` 條目格式（固定，方便日後機器整理）：

```
### {YYYY-MM-DD} [{分類: env|project|制度}] {一句話標題}
- 現象：{發生了什麼}
- 原因：{查明的原因；沒查明就寫「未查明」}
- 以後怎麼做：{一句可執行的指示}
```

## 4. 精簡規則（防止制度長成垃圾山）

- `lessons.md` 超過 **40 條**或 **300 行**：把重複主題合併、把已寫進 CLAUDE.md/制度檔的條目刪掉
  （精簡前先備份到 `backup/`）。精簡是「合併與刪除已內化的」，不是刪掉不喜歡的。
- CLAUDE.md 超過 **100 行**：把細節下放到 `docs/agents/` 對應檔，CLAUDE.md 只留路由。
- 任何制度檔超過 **250 行**：拆檔或精簡，並更新 CLAUDE.md 路由表。
- 每次精簡都是一個獨立 commit，方便回滾。

## 5. 一致性檢查點（改完任何制度檔後自問）

- [ ] 檔案裡引用的每個路徑/檔名，用 ls 或 Glob 驗過存在？
- [ ] 數字類規則（重試次數、行數上限、QG 分數）跟其他檔沒有兩套說法？
- [ ] 新規則有沒有讓弱模型無法判定的模糊詞（「適當地」「高品質」「盡量」）？有就改成可判定的。
- [ ] CLAUDE.md 路由表還指得到所有制度檔？
