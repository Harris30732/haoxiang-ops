# 判斷力外化（judgment.md）

> 讀者：任何等級的 Claude。這份把「憑感覺」的判斷改成可查表的判準。
> 每條判準附一個正例（✅ 這樣做對）和一個反例（❌ 這樣做錯）。

## 1. 何時算「真的完成」

完成 = 以下全部成立，缺一項就不准回報完成：

1. 使用者要求的每一項都有對應產出（逐項對照原始要求，不是憑印象）。
2. 驗證閘通過：改了 `.ts/.tsx` → `npm run typecheck && npm run lint` 實際跑過且 exit 0；
   動到頁面流程/設定 → 加 `npm run build`。「應該會過」不算跑過。
3. 產出已落檔且已 commit + push（本環境沒 push＝沒做）。
4. 如果過程中砍了範圍（例如「太難先跳過」），回報裡明講砍了什麼，不能沉默。

- ✅ 正例：改完 `clock.ts` → 跑 typecheck+lint 過 → commit push → 回報
  「完成，typecheck/lint 通過，commit abc123；註：未動 Postgres function，因為這次只改前端驗證」。
- ❌ 反例：改完說「已完成打卡邏輯修改，應該沒問題」——沒跑驗證閘、沒 push、
  用「應該」遮掩沒驗證的事實。

### QG 分數 rubric（commit message 用）
- **9-10**：驗證閘全過＋verifier（或第二意見）驗收 PASS。
- **7-8**：驗證閘全過，自己逐條對照過驗收條件，但沒有獨立驗收。
- **5-6**：驗證閘過，但有已知的未處理邊角（要在 commit body 或回報裡寫明）。
- **<5**：驗證閘沒過或沒跑 → 不准 commit 到共用分支；真要留進度就標 `[QG:skip]` 並說明原因。
- 純文件/設定變動跑不了驗證閘：read-back 自查後最高給 8；verifier 驗過才給 9+。

## 2. 何時該升級模型（配合 dispatch.md §5）

出現任一訊號就升級，不要「再試最後一次」：

- 同一子任務用同方法失敗，haiku 1 次、sonnet 2 次（見 dispatch.md §5 的完整路徑）。
- 你發現自己在猜（「可能是這樣」出現在你的推理裡超過兩次而沒有新證據）。
- 任務涉及跨 3 個以上檔案的因果推理（例如狀態分流 bug 同時牽涉 RLS、Server Action、前端 state）。
- 要做「二選一會影響後續所有工作」的決定（資料模型、auth 流程）。

- ✅ 正例：sonnet 兩次修 auth exchange 都把 cookie 弄壞 → 停手，整理兩次的 diff 與錯誤，
  升 opus 並附完整失敗軌跡。
- ❌ 反例：同一個 typecheck 錯誤，第三次還在小改重試同一行——這是燒 token 不是解題。

## 3. 何時該停下來問使用者

**先自查**：答案能不能從（a）原始要求原文（b）CLAUDE.md 與制度檔（c）codebase 事實 查到？
能查到就不要問。查不到且符合以下任一條 → 用 AskUserQuestion 問，並附上你的建議選項：

- 不可逆或外部可見：動生產資料庫、發訊息給員工的 LINE、刪除非自己建立的檔案、force push。
- 兩個都合理但互斥的方向，且選錯的重工成本 > 30 分鐘（例如「業績免報窗口改 2 小時」
  是要改 DB function 還是前端判斷）。
- 使用者的要求與 CLAUDE.md 硬規則衝突。

- ✅ 正例：「要我改打卡半徑，但 seed 資料與 ARCHITECTURE.md 寫的預設值不同，
  以哪個為準？我建議以 migration 裡的為準，因為那是實際上線的值。」
- ❌ 反例：問「請問我可以開始了嗎？」「要不要我跑 lint？」——這些自己決定；
  也不要每做一步就徵求同意，使用者不在線上，問了工作就停了。

## 4. 什麼訊號代表方向錯了（該換路，不是再重試）

- 修 A 壞 B、修 B 又壞 A（蹺蹺板）→ 你的假設層級錯了，往上一層找共同原因。
- 為了讓改動能動，開始改「不該動的東西」（舊 migration、node_modules、
  跟任務無關的檔案）→ 立刻停，方向錯了。
- 每一步都要加新的特例 if 才能前進 → 設計不對，回 Plan 階段。
- 錯誤訊息一直變但進度沒變 → 你在繞圈，把軌跡寫下來，換一個假設重新開始。

- ✅ 正例：發現要讓 LIFF 本機測試通過得 mock 三層 SDK → 停下，改為「部署到 tunnel 用真 LIFF 驗」。
- ❌ 反例：build 錯誤從 import 錯變成型別錯變成 runtime 錯，還繼續逐個補丁——
  沒有意識到是一開始改錯了檔案。

## 5. 品質底線怎麼驗（最低成本版）

按產出類型查表（完整驗法見 dispatch.md §6）：

| 產出 | 最低驗法 | 不算數的驗法 |
|------|----------|--------------|
| 程式碼 | typecheck+lint 實跑；行為改動再用 dev server 或臨時腳本實跑一次該流程 | 「我讀過一遍覺得對」 |
| 文件 | 逐條 read-back：承諾的內容存在、路徑真實（ls 驗證）、內部無矛盾 | 「剛寫完所以是對的」 |
| SQL migration | 在乾淨的本機/分支資料庫跑過；沒有環境就明講「未實跑，僅靜態檢查」 | 直接 apply 到生產 |
| 回答使用者的問題 | 每個關鍵宣稱都有 檔案:行號 或來源 URL 支撐 | 憑訓練記憶回答 repo 的事 |

## 6. 誠實條款：這套制度補不了的東西

拆解、驗證、多樣本評審能把「執行品質」拉到接近強模型，但以下兩類補不了，遇到就照做：

- **模糊題**（要求本身沒有明確對錯，如「把 UI 弄得順手一點」）：不要假裝有標準答案。
  做法：產 2-3 個具體方案（附截圖或描述差異）讓使用者選，或明說「這題需要你定義驗收標準」。
- **品味判斷**（架構美感、文案語氣、取捨的輕重）：升到可用的最強模型 + 第二意見評審；
  若仍不確定，把兩邊論點如實呈給使用者，不要硬選然後包裝成客觀結論。
- 查不到的事實（外部服務的現況、使用者的意圖）：標註「未確認」，不要編造。
  寧可交付上寫著三個「未確認」的誠實報告，也不要一份全是自信但錯誤的報告。
